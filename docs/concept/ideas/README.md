## 価値について

- 様々なゲームで成長曲線が急である期間に面白みを感じる点について
- 財力の観点において、財力が小さいうちの収入と財力が限度に達しようとした際の収入の価値は同額でも違うものとなる。
  - 財力の増加と収入の価値の低減が相関していると言ってよいか。
- 要素の変動により価値の変動も発生するため、仕組みとして要素と対応する軸などのグラフを作成するべき
- 価値の低減がスポイルの効果につながるため仕組みの導入により別観点の導入などにより、\
  価値が低くなった要素に対する新たな価値の創出などを行う。

## よくないてん

### 繰り返し

繰り返される演出などはユーザーが学習してしまうため悪影響のほうが大きくなる。

- コマーシャルが顕著

比較的繰り返しの耐性が高いもの。

- 音楽
  - 視覚情報が割合として多いため

#### 冗長な演出

- 演出の質がどれだけ高くとも表示される回数が多いものは冗長であると感じさせる。
  - 「ガチャ」の演出は特にその条件に当てはまる。
    - 最初のうちは演出に感心したとしても繰り返しが行われるものであるため注目しなくなっていく。
- UI のアニメーション
  - 繰り返しが多く操作の効率などを「ユーザーが学習する」場合はアニメーションのコンマ数秒が冗長となる。
    - アニメーションなしとするオプションの用意
  - アニメーションの挙動として特に問題であるのはアニメーション中は操作を受け付けない仕様の場合
    - 繰り返し行われる表示においてはその都度ユーザーは「待つ」必要が発生してしまう。
    - 繰り返しの表示でアニメーションを行うのであればユーザーに対し操作を許可したほうが、\
      アニメーションの価値を高める結果になると思われる。
- 短縮ではなく省略が望ましい場合がある。
  - ユーザーは演出などを学習してしまうことで訴求でなく悪印象につながる。
- 短縮などをユーザーの操作によらせるものはストレスを生じさせるので自動化させる。
  - 繰り返しの多い作業においては 2 つのユーザーのタイプでの悪影響が考えられる。
    - 反応速度の遅いユーザーはその操作が時間の消費につながる。
    - 反応速度の速いユーザーは繰り返しの多い価値の低い作業に対して反応するというリソースの消費を強いられることがストレスにつながる。

App Router はレイアウトの中にレイアウトが存在するため部分的なレンダリングになると考えていたが、
親子のレイアウトがそれぞれ独立したページのような構成であるというあいであ

- 親、子は同一のレイアウトでなく独立したレイアウトを使用する。
- 子が表示される際には「親の表示の上に重なるように」表示する。
  - 子は親の部品とせず、`z-index` などによる表示上の優位性を持つ形にする。

## 関係性・権限構造によるゲームシステム

[OpenFGA](https://openfga.dev/) のような ReBAC (Relationship-Based Access Control) モデルを、ユーザー認可でなくゲーム内の関係性表現に転用できないか。

- type / relation / tuple による関係グラフをゲーム内の所有・アクセス条件などに応用
  - 例: アイテムの所有関係、鍵と扉の解錠条件、エリアの解放条件
- 継承・階層構造 (グループ経由の権限など) をスキルツリーや装備条件、エリア解放条件の前提関係に応用できないか
- 想定される向き不向き
  - 向く: インベントリ所有権、クエスト進行条件、エリア解放条件、ギルド/パーティ権限など、変化頻度の低い静的〜準静的な関係性
  - 向かない: リアルタイム戦闘判定や物理演算など、フレーム単位で発生する状態変化
- 詰め切れていない点: サーバ常時起動が前提となる構成のため、ローカル完結のゲームにどこまで適用するかは要検討
